# Step 9: frontend-export-flow

## 읽어야 하는 파일

먼저 아래 파일들을 읽고 프로젝트의 아키텍처와 설계 의도를 파악하라:

- `AGENTS.md`
- `docs/PRD.md`
- `docs/ARCHITECTURE.md`
- `docs/UI_GUIDE.md`
- `phases/0-mvp/index.json`
- `frontend/`

이전 frontend 화면과 export API client를 읽고 사용자-facing 용어 규칙을 유지하라.

## 작업

주문 기록 선택, 주문 미리보기, 주문 완료/다운로드 흐름을 구현한다. 코드 내부와 API path에서는 export 용어를 사용할 수 있지만, 사용자-facing 화면 문구에서는 주문, 기록 선택, 주문 미리보기, 주문 완료, 다운로드 용어를 사용한다.

화면:
- `/export`
  - 기록 목록에서 주문 가능한 항목을 체크 선택할 수 있게 한다.
  - 행 전체 클릭과 Space 키 토글을 지원한다.
  - 잠금 기록은 주문 대상에서 비활성 처리한다.
  - 선택 개수와 주문 신청 버튼 상태를 명확히 보여준다.
  - 공개 완료된 기록이 없으면 "둘 다 답한 기록만 주문할 수 있어요." 안내를 표시하고 버튼을 비활성화한다.
- 주문 미리보기 상태
  - 사용자가 신청한 날짜, 질문, 답변 목록을 최종 확인한다.
  - 실제 다운로드에 포함될 내용만 보여준다.
  - 주요 버튼은 "주문 완료", 보조 버튼은 "다시 선택" 또는 "취소"를 사용한다.
- 주문 완료 상태
  - 주문이 완료되었음을 짧게 알려준다.
  - "JSON 다운로드", "텍스트 다운로드" 버튼을 제공한다.
  - 완료 후에도 선택된 기록 요약을 보여준다.
  - 주문 완료 전에는 다운로드 버튼을 노출하지 않는다.

UX 요구:
- 모바일 하단 액션은 필요한 경우 sticky 영역으로 제공하되 콘텐츠를 가리지 않게 한다.
- 체크박스와 주요 버튼은 최소 44px 높이의 터치 영역을 확보한다.
- 상태는 색상만으로 표현하지 않고 텍스트 배지와 안내 문구를 함께 사용한다.
- 결제나 실물 제작으로 오해할 수 있는 문구를 쓰지 않는다.

테스트:
- 주문 가능한 기록 선택/해제, 아무 항목도 선택하지 않은 상태, 잠금 항목 비활성을 테스트한다.
- 주문 신청 -> 미리보기 -> 완료 -> 다운로드 버튼 노출 흐름을 테스트한다.
- 완료 전 다운로드 버튼이 없는지 테스트한다.
- 사용자-facing 문구에 export 단어가 노출되지 않는지 핵심 화면 기준으로 테스트한다.

## Acceptance Criteria

```bash
cd frontend && npm run build
cd frontend && npm run lint
cd frontend && npm run test
```

## 검증 절차

1. AC 커맨드를 실행한다.
2. 사용자-facing 문구에서 export 단어가 보이지 않는지 확인한다.
3. 주문 완료 전 다운로드 버튼이 렌더링되지 않는지 확인한다.
4. 성공하면 `phases/0-mvp/index.json`의 step 9를 `"completed"`로 바꾸고 `summary`에 export/order UI 산출물을 한 줄로 요약한다.

## 금지 사항

- backend 파일을 수정하지 마라. 이유: frontend 주문 흐름만 담당한다.
- 결제, 배송, PDF, 실물 인쇄 주문을 구현하지 마라. 이유: MVP 제외 사항이다.
- 카드 안에 큰 카드를 중첩하지 마라. 이유: UI_GUIDE의 카드 규칙 위반이다.
