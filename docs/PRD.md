# PRD: 하루 한 질문 교환일기

## 목표
커플이 매일 하나의 질문에 각자 답하고, 둘 다 답한 뒤 서로의 답변을 열람하며, 쌓인 답변 중 원하는 기록만 골라 주문 신청, 미리보기 확인, 주문 완료 후 JSON/text로 다운로드할 수 있게 한다. 주문 예약 단계의 주문은 나중에 이어서 완료하거나 취소할 수 있고, 완료된 주문은 주문 내역에서 저장된 스냅샷을 다시 확인하거나 다운로드하거나 삭제할 수 있다.

## 제품 원칙
- 매일 돌아올 이유가 분명해야 한다. 첫 화면은 오늘의 질문과 답변 상태다.
- 답변은 서로에게 영향을 덜 받도록 동시 공개한다.
- 사용자의 사적인 기록은 기본적으로 닫혀 있어야 한다.
- MVP는 데이터와 루틴 검증이 목표다. 결제, 배송, PDF, 실물 제작은 넣지 않는다.
- "주문"은 MVP에서 export 주문을 뜻한다. 사용자가 실물 주문으로 오해하지 않게 UI 문구를 명확히 한다.

## 사용자
- 연인 또는 커플
- 장거리 연애처럼 매일의 대화 루틴이 필요한 커플
- 관계의 기억, 취향, 감정, 대화를 조용히 기록하고 싶은 사용자
- 나중에 기념일 편지, 책, 카드, 앨범 같은 결과물로 확장할 답변 데이터를 모으고 싶은 사용자

## MVP 성공 기준
- 사용자가 이메일/비밀번호로 가입하고 다시 로그인할 수 있다.
- 한 사용자가 초대 코드를 만들고, 다른 사용자가 그 코드로 커플에 참여할 수 있다.
- 커플은 매일 하나의 질문을 받고, 같은 날짜에는 같은 질문을 다시 본다.
- 한 명만 답변한 경우 상대 답변 내용은 노출되지 않는다.
- 둘 다 답변하면 양쪽 답변이 공개된다.
- 사용자가 과거 기록을 선택해 주문 신청, 미리보기, 주문 완료, 다운로드까지 끝낼 수 있다.
- 사용자가 주문 내역에서 주문 예약 단계의 주문을 재개하고 완료된 주문의 저장 스냅샷을 다시 확인, 다운로드, 삭제할 수 있다.
- 다른 커플의 데이터에 접근할 수 없다.

## 핵심 기능
1. 이메일/비밀번호 회원가입 및 로그인
2. 초대 코드 기반 커플 연결
3. Asia/Seoul 날짜 기준 매일 하나의 질문 제공
4. 오늘 질문에 대한 답변 작성 및 수정
5. 둘 다 답변한 뒤 서로의 답변을 공개하는 동시 공개
6. 날짜별 질문과 답변 기록 열람
7. 원하는 기록을 체크 선택해 주문 신청하는 기능
8. 주문 확인 전 선택한 기록을 미리보기로 검토하는 기능
9. 주문 완료 후 JSON/text 결과를 다운로드하는 기능
10. 주문 내역에서 주문 예약 단계 재개와 완료 주문 상세 보기, 재다운로드, 삭제 기능

## 핵심 사용자 흐름
1. 사용자가 이메일과 비밀번호로 가입하거나 로그인한다.
2. 커플 공간이 없으면 초대 코드를 생성하거나 상대가 만든 초대 코드를 입력한다.
3. 커플 연결이 완료되면 오늘의 질문 화면으로 이동한다.
4. 사용자는 오늘 질문에 답변을 작성한다.
5. 상대가 아직 답하지 않았으면 상대 답변은 잠금 상태로 보인다.
6. 둘 다 답변하면 양쪽 답변이 공개된다.
7. 기록 화면에서 과거 질문과 답변을 확인한다.
8. 주문 기록 선택 화면에서 원하는 날짜의 기록을 체크하고 주문 신청을 누른다.
9. 주문 확인 전 화면에서 선택한 질문, 날짜, 답변 내용을 미리보기로 검토한다.
10. 사용자가 확인하면 주문이 완료된다.
11. 주문 완료 화면에서 선택된 항목만 포함한 JSON/text 결과를 다운로드한다.
12. 사용자가 주문 내역으로 돌아오면 주문 예약 단계의 주문은 이어서 완료하거나 취소하고, 완료된 주문은 같은 스냅샷을 다시 확인하거나 다운로드하거나 삭제한다.

## 사용자 상태와 라우팅
- 미로그인 사용자가 보호 화면(`/today`, `/diary`, `/export`)에 접근하면 `/login`으로 이동한다.
- 로그인했지만 커플이 없는 사용자는 `/couple`로 이동한다.
- 초대 코드를 생성한 사용자는 상대가 참여하기 전까지 상대 초대 대기 화면을 본다.
- 초대 코드로 커플 연결이 완료되면 `/today`로 이동한다.
- 이미 로그인했고 커플 연결이 완료된 사용자가 앱 루트에 접근하면 `/today`로 이동한다.
- 세션이 만료되면 현재 작업을 중단하고 `/login`으로 이동하되, 가능하면 "다시 로그인해주세요." 안내를 표시한다.

## 사용자 문구와 내부 용어
- 사용자-facing 화면, 버튼, 빈 상태 문구에서는 `export`라는 단어를 쓰지 않는다.
- 사용자-facing 용어는 주문, 기록 선택, 주문 미리보기, 주문 완료, 다운로드를 사용한다.
- 개발 내부, API path, 도메인 모델, 파일명에서는 `export` 용어를 유지한다.
- MVP의 "주문"은 결제나 실물 제작이 아니라 JSON/text 다운로드를 위한 기록 묶음 생성이다.

## 화면별 요구사항
### 회원가입
- 입력값: 이메일, 비밀번호, 표시 이름
- 이메일은 중복될 수 없다.
- 이메일은 기본 이메일 형식을 만족해야 한다.
- 비밀번호는 8자 이상 72자 이하로 한다.
- 표시 이름은 1자 이상 20자 이하로 한다.
- 비밀번호는 서버에 평문으로 저장하지 않는다.
- 가입 성공 후 로그인 상태가 되거나 로그인 화면으로 이동한다. MVP 기본값은 가입 후 로그인 상태다.

### 로그인
- 입력값: 이메일, 비밀번호
- 로그인 성공 시 서버 세션이 생성된다.
- 로그인 실패 시 이메일 존재 여부를 노출하지 않는 일반 오류를 보여준다.
- 이미 로그인한 사용자는 오늘 질문 또는 커플 연결 화면으로 이동한다.

### 커플 연결
- 커플이 없는 사용자는 초대 코드 생성 또는 초대 코드 입력 중 하나를 선택한다.
- 초대 코드는 24시간 동안 유효하다.
- 초대 코드는 한 번 성공적으로 사용되면 재사용할 수 없다.
- 커플은 최대 2명만 허용한다.
- 이미 커플에 속한 사용자는 다른 커플에 참여할 수 없다.
- 초대 코드를 생성한 사용자는 코드, 만료 시간, 복사 버튼, 상대 참여 대기 상태를 볼 수 있어야 한다.
- 상대가 초대 코드로 참여하면 대기 화면은 오늘 질문 화면으로 전환된다.

### 오늘 질문
- 로그인했고 커플에 속한 사용자만 접근할 수 있다.
- Asia/Seoul 날짜 기준으로 오늘의 질문을 조회한다.
- 오늘 질문이 아직 배정되지 않았다면 서버가 질문 풀에서 하나를 배정한다.
- 같은 커플과 같은 날짜에는 항상 같은 DailyQuestion을 반환한다.
- 질문 풀을 모두 사용한 경우 MVP에서는 오래된 질문 재사용을 허용한다.
- 질문 재사용 순서는 커플별 누적 DailyQuestion 개수를 active 질문 개수로 나눈 나머지 인덱스를 사용한다.

## 오늘 질문 상태 모델
- NOT_ANSWERED: 내가 아직 답하지 않았고 상대 답변 내용도 볼 수 없는 상태
- MY_ANSWERED_PARTNER_WAITING: 나는 답했고 상대 답변을 기다리는 상태
- PARTNER_ANSWERED_ME_WAITING: 상대는 답했고 나는 아직 답하지 않은 상태. 상대 답변 내용은 잠금 처리한다.
- BOTH_ANSWERED: 둘 다 답해 양쪽 답변이 공개된 상태
- PARTNER_ANSWERED_ME_WAITING 문구는 "상대가 답변을 마쳤어요. 내 답변을 남기면 함께 열려요."를 기본으로 한다.
- MY_ANSWERED_PARTNER_WAITING 문구는 "내 답변은 저장됐어요. 상대가 답하면 함께 열려요."를 기본으로 한다.
- 상태 문구는 답변 내용을 추측하게 만들지 않아야 한다.

### 답변 작성/수정
- 사용자는 오늘 질문에 답변을 작성할 수 있다.
- 사용자는 자신의 답변만 수정할 수 있다.
- 답변은 빈 문자열로 저장할 수 없다.
- MVP 기본 제한은 1자 이상 2000자 이하로 한다.
- 답변 수정 이력은 MVP에서 저장하지 않는다.

## 답변 작성 UX
- MVP는 자동 저장을 지원하지 않는다. 사용자는 저장 버튼을 눌러야 답변이 저장된다.
- 답변 입력 중 저장하지 않고 페이지를 벗어나려 하면 브라우저 기본 이탈 경고 또는 앱 내부 확인 모달을 보여준다.
- 저장 버튼은 답변이 비어 있거나 2000자를 초과하면 비활성화한다.
- 저장 요청 중에는 저장 버튼을 비활성화해 중복 클릭을 막는다.
- 저장 성공 후 "저장됐어요." 같은 짧은 상태 피드백을 표시한다.
- 저장 실패 시 입력 내용은 화면에 유지하고 오류 안내를 표시한다.

### 기록 열람
- 날짜 내림차순으로 기록을 보여준다.
- 각 기록은 날짜, 질문, 내 답변 상태, 상대 답변 공개 여부를 포함한다.
- 둘 다 답변한 기록은 양쪽 답변 내용을 보여준다.
- 한 명만 답변한 기록은 상대 답변 내용을 잠금 처리한다.

## 공개 규칙
- 내 답변은 작성 후 바로 확인할 수 있다.
- 상대 답변은 상대가 답하기 전까지 잠금 상태로 표시한다.
- 둘 다 답한 질문만 서로의 답변 내용을 볼 수 있다.
- 서버는 잠금 상태의 상대 답변 내용을 응답에 포함하지 않는다.
- 주문 미리보기와 다운로드도 공개 가능한 답변만 포함한다.

## 주문 흐름
1. 주문 신청: 사용자가 기록 목록에서 원하는 항목을 체크하고 주문 신청을 시작한다.
2. 주문 확인 전 미리보기: 서버는 현재 사용자가 접근 가능한 공개 답변만 모아 미리보기 데이터를 생성하고 주문 상태를 PREVIEWED로 바꾼다. 사용자는 포함될 날짜, 질문, 답변을 최종 확인한다.
3. 주문 완료: 사용자가 미리보기를 확인하면 export 주문이 완료되고, 서버는 완료 시점의 JSON/text 결과를 스냅샷으로 저장한다.
4. 다운로드: 주문 완료 화면에서 JSON 파일과 text 파일을 내려받을 수 있다.
5. 주문 내역: PREVIEWED 주문은 "주문 예약 단계"로 표시하고 미리보기를 다시 열어 완료 또는 취소할 수 있다. COMPLETED 주문은 "주문 완료"로 표시하고 저장된 스냅샷 상세 보기, 다운로드, 삭제를 할 수 있다.

## 주문 상태
- REQUESTED: 사용자가 기록을 선택하고 주문 신청을 만든 상태
- PREVIEWED: 주문 확인 전 미리보기 데이터가 생성되어 사용자가 최종 확인할 수 있는 주문 예약 단계
- COMPLETED: 사용자가 최종 확인해 저장 스냅샷과 다운로드 가능한 결과가 생성된 주문 완료 상태
- CANCELLED: 사용자가 미리보기 단계에서 주문을 취소한 상태

## 주문 규칙
- 주문 신청에는 하나 이상의 기록이 필요하다.
- 주문 항목은 현재 사용자의 커플 기록만 포함할 수 있다.
- 잠금 상태 기록은 주문 신청 대상에서 비활성 처리한다.
- 둘 다 답한 기록만 주문 대상이 될 수 있다.
- 주문 완료 전에는 다운로드 버튼과 다운로드 API를 사용할 수 없다.
- 주문 완료 후에는 선택 항목과 다운로드 내용을 바꾸지 않는다. 바꾸려면 새 주문을 만든다.
- MVP에서는 외부 파일 스토리지를 쓰지 않는다.
- 주문 완료 시점의 JSON/text 결과는 DB에 스냅샷으로 저장해 이후 다운로드가 같은 내용을 반환하도록 한다.
- 주문 내역은 현재 세션 사용자의 커플 주문만 보여준다.
- 주문 내역의 기본 목록은 PREVIEWED와 COMPLETED 주문만 보여준다.

## 빈 상태와 비활성 상태
- 기록이 없으면 오늘 질문으로 이동할 수 있는 안내를 표시한다.
- 공개 완료된 기록이 없으면 주문 신청 버튼을 비활성화한다.
- 잠금 기록만 있으면 "둘 다 답한 기록만 주문할 수 있어요." 안내를 표시한다.
- 주문 기록 선택에서 아무 항목도 선택하지 않으면 주문 신청 버튼을 비활성화한다.
- 오늘 질문, 기록 목록, 주문 미리보기 로딩 중에는 skeleton 또는 짧은 대기 문구를 표시한다.
- API 요청 중 같은 액션 버튼은 비활성화해 중복 요청을 막는다.
- 주문 완료 화면은 사용자가 다운로드를 놓쳐도 다시 누를 수 있도록 선택 요약과 다운로드 버튼을 유지한다.
- 주문 내역이 없으면 기록 선택으로 주문 신청을 시작하라는 안내를 표시한다.

## Export 결과
### JSON
- 파일명 형식: `couple-diary-{exportRequestId}.json`
- 인코딩: UTF-8
- 날짜: `YYYY-MM-DD`
- 정렬: 날짜 오름차순
- 포함 필드: exportRequestId, coupleId, exportedAt, entries

### Text
- 파일명 형식: `couple-diary-{exportRequestId}.txt`
- 인코딩: UTF-8
- 정렬: 날짜 오름차순
- 각 항목은 날짜, 질문, 답변자 표시 이름, 답변 본문을 포함한다.
- 줄바꿈은 `\n`을 사용한다.

## 예외 케이스
- 미로그인 사용자가 보호 화면에 접근하면 로그인 화면으로 이동한다.
- 커플이 없는 사용자가 오늘 질문이나 기록에 접근하면 커플 연결 화면으로 이동한다.
- 로그인 실패 시 이메일 존재 여부를 알 수 없는 일반 오류를 보여준다.
- 회원가입 입력값이 잘못되면 필드별 오류를 보여준다.
- 만료되었거나 사용된 초대 코드는 사용할 수 없다.
- 이미 2명이 연결된 커플에는 참여할 수 없다.
- 이미 커플에 속한 사용자는 다른 초대 코드로 참여할 수 없다.
- 같은 사용자가 같은 DailyQuestion에 중복 답변을 생성하려고 하면 기존 답변 수정으로 처리하거나 오류를 반환한다. MVP 기본값은 기존 답변 수정이다.
- 질문 seed가 비어 있으면 오늘 질문 API는 서버 설정 오류를 반환한다.
- 선택한 주문 항목 중 접근 불가 항목이 있으면 주문 신청을 실패시킨다.
- 주문이 COMPLETED가 아니면 다운로드 요청을 거부한다.
- 주문 완료 후 원본 답변이 수정되어도 기존 주문 다운로드 내용은 바뀌지 않는다.
- 주문 완료 후 항목이나 내용은 수정할 수 없으며, 바꾸려면 새 주문을 만든다.
- 주문 완료 후 주문 내역에서 완료 주문을 삭제할 수 있다.
- 다른 커플의 주문 목록이나 주문 상세에 접근할 수 없다.
- 지원하지 않는 다운로드 format은 validation error로 처리한다.
- 서버 오류가 발생하면 내부 오류 내용 대신 재시도 안내 메시지를 보여준다.

## 에러 UX 요구사항
- 입력값 오류는 가능한 경우 해당 필드 아래에 표시한다.
- 인증 만료 또는 미로그인은 로그인 화면으로 이동한다.
- 권한 없음은 "접근할 수 없는 기록입니다."처럼 짧게 표시한다.
- 상태 충돌은 데이터를 다시 불러오도록 안내한다.
- 409 상태 충돌은 관련 데이터를 refetch해 최신 상태로 복구한다.
- 서버 오류는 사용자의 잘못으로 보이지 않게 표현한다.
- 500 서버 오류 문구는 "잠시 후 다시 시도해주세요."로 통일한다.
- 어떤 화면에서도 stack trace, SQL error, 내부 exception 이름을 노출하지 않는다.

## 개인정보와 보안 요구사항
- 비밀번호는 BCrypt로 해시한다.
- 세션에는 userId만 저장한다.
- 모든 보호 API는 현재 세션 사용자 기준으로 커플 소유권을 검증한다.
- 잠금 상태의 상대 답변 content는 서버 응답에 포함하지 않는다.
- 로그에 비밀번호, 답변 본문, 세션 ID를 남기지 않는다.

- 사용자는 "서비스를 쓰고 있다"보다 "둘만의 기록장을 열었다"는 느낌을 받아야 한다.

## Acceptance Criteria
- 회원가입, 로그인, 로그아웃, 현재 사용자 조회가 동작한다.
- 초대 코드 생성, 만료, 사용 완료 처리가 동작한다.
- 두 명의 사용자가 하나의 커플로 연결된다.
- 세 번째 사용자는 같은 커플에 참여할 수 없다.
- 오늘 질문은 같은 커플과 같은 날짜에 동일하다.
- 답변 작성 전, 한 명 답변 후, 둘 다 답변 후 상태가 명확히 구분된다.
- 잠금 상태에서 상대 답변 content가 API 응답에 없다.
- 기록 목록은 날짜 내림차순으로 조회된다.
- 주문은 REQUESTED, PREVIEWED, COMPLETED, CANCELLED 상태를 가진다.
- COMPLETED 전에는 다운로드할 수 없다.
- PREVIEWED 주문은 주문 내역에서 주문 예약 단계로 표시하고 미리보기를 다시 열어 완료 또는 취소할 수 있다.
- COMPLETED 주문은 주문 내역에서 주문 완료로 표시하고 저장된 스냅샷 상세 보기, JSON/text 재다운로드, 삭제를 할 수 있다.
- JSON/text 다운로드 결과는 같은 entries를 같은 순서로 포함한다.
- 미로그인 사용자가 `/today`, `/diary`, `/export`에 접근하면 로그인으로 이동한다.
- 로그인했지만 커플이 없는 사용자는 `/couple`로 이동한다.
- 초대 코드를 만든 사용자는 상대 참여 전까지 대기 화면을 본다.
- 상대가 먼저 답변한 상태에서도 상대 답변 content는 보이지 않고 "상대가 답변을 마쳤어요. 내 답변을 남기면 함께 열려요." 상태만 보인다.
- 공개 완료된 기록이 없으면 주문 신청 버튼이 비활성화된다.
- 모바일에서 주요 버튼과 체크박스 행은 터치하기 충분한 크기다.
