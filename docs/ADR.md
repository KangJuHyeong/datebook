# Architecture Decision Records

## 철학
로컬 MVP를 빠르게 검증하되, 커플의 사적인 기록을 다루는 제품답게 인증, 권한, 답변 공개 규칙은 처음부터 명확히 지킨다. 외부 의존성은 최소화하고, 매일 쓰는 기록 경험과 데이터 export 가능성을 우선한다.

---

### ADR-001: Next.js App Router 선택
**결정**: 프론트엔드는 Next.js App Router, React, TypeScript, Tailwind CSS로 구현한다.

**이유**: 페이지 라우팅, 레이아웃 구성, 클라이언트 인터랙션을 빠르게 만들 수 있고 TypeScript와 Tailwind CSS 조합으로 MVP UI를 안정적으로 구성할 수 있다.

**트레이드오프**: Next.js API Route에 비즈니스 로직을 넣지 않는다. 프론트엔드는 Spring Boot API client 역할에 집중하므로 API 서버와의 CORS, 세션 쿠키 설정을 함께 관리해야 한다.

### ADR-002: Spring Boot API 서버와 MySQL 선택
**결정**: 백엔드는 Spring Boot, Java, Spring Data JPA를 사용하고 데이터베이스는 MySQL을 사용한다.

**이유**: 사용자, 커플, 질문, 답변, export 이력은 관계형 모델로 표현하기 적합하다. Spring Data JPA는 CRUD와 도메인 규칙 구현에 충분하며, MySQL은 로컬 개발과 추후 운영 전환이 쉽다.

**트레이드오프**: 프론트엔드와 백엔드가 분리되어 로컬 실행 프로세스가 둘로 나뉜다. CORS와 세션 쿠키 설정을 정확히 맞춰야 한다.

### ADR-003: 세션 기반 이메일/비밀번호 인증 선택
**결정**: MVP 인증은 이메일/비밀번호 로그인과 Spring 서버 세션으로 구현한다.

**이유**: 소셜 로그인이나 JWT 갱신 흐름 없이도 로컬 MVP에서 충분한 인증 경험을 만들 수 있다. 서버 세션에 userId를 저장하면 보호 API에서 현재 사용자를 일관되게 식별할 수 있다.

**트레이드오프**: 프론트엔드 요청은 credentials 포함이 필수이며, 로컬 CORS 설정이 필요하다. 모바일 앱이나 다중 도메인 배포를 고려할 때는 인증 전략을 다시 검토할 수 있다.

### ADR-004: 초기 질문은 AI seed를 DB에 저장하고 런타임 AI는 제외
**결정**: 초기 질문은 운영자가 AI로 생성한 뒤 seed 데이터로 MySQL에 저장한다. MVP 런타임에서는 AI API를 호출하지 않는다.

**이유**: 질문 품질을 사전에 검수할 수 있고, 비용과 장애 지점을 줄일 수 있다. 매일 질문 제공이라는 핵심 경험은 정적 질문 풀만으로도 검증 가능하다.

**트레이드오프**: 사용자별 개인화 질문은 MVP에서 제공하지 않는다. 사용자 직접 질문 추가나 AI 일일 생성 기능은 v2로 미룬다.

### ADR-005: 동시 공개 답변 모델 선택
**결정**: 둘 다 답변한 뒤 서로의 답변을 공개한다. 한 명만 답변한 상태에서는 상대 답변을 잠금 상태로 표시한다.

**이유**: 상대 답변을 먼저 보고 영향을 받는 문제를 줄이고, 교환일기의 공정함과 설렘을 유지할 수 있다.

**트레이드오프**: 한 명이 답하지 않으면 상대 답변을 바로 볼 수 없어 즉시 만족감은 줄어든다. 대신 잠금 상태와 대기 문구를 통해 흐름을 명확히 안내한다.

### ADR-006: 주문은 신청, 미리보기, 완료 후 JSON/text 다운로드로 구성
**결정**: MVP의 주문 기능은 사용자가 체크 선택한 기록으로 주문 신청을 만들고, 주문 확인 전 미리보기를 거친 뒤, 주문 완료 상태에서만 JSON과 plain text를 다운로드하는 기능으로 제한한다.

**이유**: 단일 export 버튼보다 사용자가 포함될 기록을 확인하기 쉽고, 추후 PDF, 책, 카드, 인쇄 주문으로 확장할 때도 주문 상태 모델을 그대로 키울 수 있다.

**트레이드오프**: MVP에서는 결제, 배송, 인쇄물 미리보기 같은 커머스 경험을 제공하지 않는다. 미리보기는 JSON/text에 포함될 기록 확인 용도이며, 사용자가 결과물 범위를 오해하지 않도록 주문 완료 화면에서 다운로드 가능한 형식을 명확히 표시한다.

### ADR-007: Asia/Seoul 날짜 기준 선택
**결정**: 하루 질문 배정 기준 날짜는 Asia/Seoul LocalDate로 계산한다.

**이유**: 초기 사용자를 한국어 사용자로 가정하고, "하루에 하나"라는 제품 경험은 사용자의 생활 날짜와 맞아야 한다.

**트레이드오프**: 해외 거주 커플이나 서로 다른 시간대 사용자는 MVP에서 세밀하게 지원하지 않는다. 추후 사용자별 시간대 설정이 필요해질 수 있다.

**보충**: created_at, updated_at 같은 timestamp는 UTC 기준으로 저장한다. 사용자의 "오늘"을 결정하는 question_date만 Asia/Seoul 기준으로 계산한다.

### ADR-008: 커플은 MVP에서 정확히 최대 2명만 허용
**결정**: 한 사용자는 하나의 커플에만 속하고, 한 커플은 최대 2명의 멤버만 가진다.

**이유**: 동시 공개, 상대 답변 잠금, 주문 export 모델이 2인 관계에 최적화되어 있다. MVP의 핵심 가설도 연인 간 교환일기다.

**트레이드오프**: 다자 관계, 가족, 친구 그룹, 여러 커플 기록은 지원하지 않는다. 나중에 그룹형 제품으로 확장하려면 공개 규칙과 데이터 모델을 다시 설계해야 한다.

### ADR-009: 질문 배정은 sort_order 기반 순환으로 시작
**결정**: DailyQuestion이 없는 날짜에는 active 질문을 sort_order, id 오름차순 기준으로 순환 배정한다. 선택 인덱스는 해당 커플의 기존 DailyQuestion 개수 modulo active 질문 개수다.

**이유**: 런타임 AI나 복잡한 추천 없이도 매일 질문 제공을 안정적으로 검증할 수 있다. seed 질문만 관리하면 운영이 쉽다.

**트레이드오프**: 커플별 개인화나 관계 단계별 추천은 제공하지 않는다. 질문이 반복될 수 있으며, v2에서 카테고리와 개인화 규칙을 추가할 수 있다.

### ADR-010: 완료된 주문 결과는 DB 스냅샷으로 저장
**결정**: MVP에서는 외부 파일 스토리지를 쓰지 않는다. 대신 주문 완료 시점의 JSON/text payload를 export_requests에 스냅샷으로 저장한다.

**이유**: 주문 완료 후 다운로드 결과가 바뀌지 않아야 사용자가 신뢰할 수 있다. 외부 파일 스토리지를 도입하지 않아도 DB 스냅샷만으로 같은 결과를 반복 다운로드할 수 있다.

**트레이드오프**: 답변 본문이 export_requests에도 중복 저장된다. 개인정보 보관 범위가 늘어나므로 완료된 주문 payload도 답변과 같은 수준으로 보호해야 한다.

### ADR-011: QueryDSL은 초기 필수 의존성에서 제외
**결정**: 초기 구현은 Spring Data JPA repository와 JPQL 중심으로 시작하고, 복잡한 동적 조회가 필요할 때 QueryDSL을 도입한다.

**이유**: MVP의 주요 조회는 커플, 날짜, 상태 기준으로 단순하다. QueryDSL 설정과 generated source 관리는 초기 속도를 늦출 수 있다.

**트레이드오프**: 기록 필터, 기간 검색, 태그, 고급 주문 조건이 생기면 repository 코드가 복잡해질 수 있다. 그 시점에는 QueryDSL 도입을 ADR로 다시 기록한다.

### ADR-012: 잠금 답변은 API 응답에서 content를 제거
**결정**: 상대가 아직 답하지 않은 상태의 답변 content는 null이나 마스킹 문자열로 내려주지 않고, 응답 객체 자체를 LOCKED 상태로 내려준다.

**이유**: 클라이언트 버그나 화면 실수로 잠긴 답변이 노출되는 위험을 줄인다. 공개 규칙은 UI가 아니라 서버 응답 계약에서 보장해야 한다.

**트레이드오프**: 클라이언트는 partnerAnswer가 공개 답변인지 LOCKED 상태인지 분기해야 한다. 대신 보안과 제품 규칙이 더 명확해진다.
